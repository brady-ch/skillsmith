# Problem Solving & Decision Making

- Keep the simplest valid explanation.
- Do not let overthinking or too much information stall the decision.
- Stop once the answer is good enough.
- Focus on the vital few.
- Use top-down, bottom-up, or divide-and-conquer strategies.
- Make the goal visible and build iteratively with feedback loops.
- Shift validation earlier.
- First make it work, then make it right, then make it pretty.
- Aim for simple, correct, consistent, and complete.

# Features

- Don't build what is not needed.
- Simpler is often better than more.
- Aim for something usable first, then something marketable.
- Keep value and return in view.

# System Design

- Keep it simple and avoid duplication.
- Be strict in what you produce and tolerant in what you accept.
- Fail fast.
- Grow from a simple working core.
- Accept that some complexity is unavoidable.
- Make responsibilities, boundaries, coupling, and interfaces explicit.

# Performance

- Mechanical sympathy
- Latency numbers

# Concurrency

- Distributed systems need clear delivery guarantees.
- Exactly-once is hard.
- At-least-once works with idempotence.
- At-most-once is fine when loss is acceptable.

# Testing

- Good tests are automated, specific, deterministic, isolated, composable, fast, readable, cheap to write, behavior-focused, and predictive.

# Time Management

- Prioritize urgent vs. important work.
- Watch for time expanding to fill the slot.
- Avoid overemphasizing trivial issues.

# Collaboration

- Make ownership clear.
- Communicate with BLUF: lead with the conclusion, then add details.
